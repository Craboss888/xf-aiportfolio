Act 3 · Blind Review

    

      
让 AI 反对你，靠制度不靠人设

      
通常的做法是给一个模型套几个「专家人设」——但它们看着同一份上下文、也看得见你的偏好，聊两句就集体同意你。laoban.ai 把「分歧」做成结构：

      
    
        
  • 盲评隔离：四位董事（CFO / 市场 / 客户代表 / 法务）各自独立调用，只看得到自己那份证据——财务只看报表、客户代表只看用户评论，看不到彼此，也看不到你的偏好。
    • 
        
  • 专属证据管道：分歧来自「看到的信息不同」，而不是提示词里写的性格不同。
    • 
        
  • 唱反调先生：一个不掌握任何数据的特殊席位，专职攻击问题本身的框架，以「提出最强反对」论功。
    •

关键子系统：自动打码（最复杂）

      
一个多阶段图像处理 pipeline，是整个项目最花时间的地方。

      
三步走

      
    
        
  • 检测：叠三个现成模型——WD14 判穿衣/裸露、booru_yolo 定位部位、NudeNet 框敏感区。
    • 
        
  • 分割：用 SAM2 把检测框转成精确像素轮廓（本地 M4 约 1 秒/张）。
    • 
        
  • 贴码：按规则（角度 / 大小 / 不遮脸 / 不重叠）贴遮挡。
    • 
      

      
怎么从 30% 提到 70%

      
    
        
  • 纯靠 AI 自动打，一开始能用率只有 30%。
    • 
        
  • 我亲手标注 100 张当「标准答案」，让系统学我的打码规律（角度、覆盖度）。
    • 
        
  • 改成双 Agent：一个负责打码、一个负责评价（像素级校验覆盖率），不达标自动迭代。
    • 
        
  • 踩坑：让子 Agent「开放式」代评不可靠，换成定向验证（「这块遮住了吗？」）才准 → 通过率提到 70%。

关键子系统：动态壁纸打包（逆向 .livp）

      
iPhone 的实况照片（.livp）格式没有公开文档，我是一点点反向工程试出来的。

      
打包链路

      
    
        
  • 静态图 → 可灵生成 5 秒视频。
    • 
        
  • 因为 iOS 只播放 Live Photo 时长的 25–50%，要用 FFmpeg 补帧到 20 秒再加速，才能完整呈现。
    • 
        
  • 配对成 HEIC + MOV、打包成 .livp，传百度云分发。
    • 
      

      
难点全在二进制细节

      
    
        
  • HEIC 与 MOV 必须同一个 UUID；文件里 56 字节的 comment 要动态算偏移。
    • 
        
  • 光是让 iPhone 认出这是「实况」，那两天我就疯狂做了 80+ 轮测试才跑通。
    • 
      

      
FFmpeg 补帧.livp 逆向HEIC + MOV 同 UUID

    

  


  
  

    

      
我从这个项目里学到的
